Dataset: TinyStories

Why TinyStories?

We need a dataset that is (1) large enough for the model to learn form and meaning of English, but (2) small enough to train under $10. TinyStories sits at this intersection — 2 million short stories using vocabulary a 3-4 year old can understand.

The paper "How small can language models be and still speak coherent English?" showed that even 1M-parameter models can learn coherent English from this dataset, because it preserves subject-verb-object structure and semantic meaning without unnecessary complexity.

What the dataset preserves

Form: Subject → Verb → Object grammar structure.
Meaning: No nonsensical sentences like "Blue electrons eat fish."
Patterns: Enough repeated structures for the model to learn language rules.

Key Numbers

2 million stories
~200M tokens total
~150M training tokens (80%)
~50M validation tokens (20%)

Tokenization: From Characters to Harmony

Why tokenization matters

AI cannot understand words. We must convert text into numbers (token IDs). The how of this conversion dramatically affects model quality, vocabulary size, and training cost.

Three approaches compared

Three tokenization strategies — Subword (BPE) is the best of both worlds

Harmony Tokenizer (O200K Harmony)

Why Harmony over plain O200K?

Plain O200K base handles text well. But modern models need to understand conversations, tool calls, and reasoning chains. Harmony adds special tokens for these capabilities so the model can learn role-based interactions during pre-training itself — not just during fine-tuning.

Token TypeExamplesPurpose
Chat Tokens<|start|>, <|end|>, <|message|>Mark conversation boundaries
Role Tokenssystem, user, assistantIdentify who is speaking
Channel Tokens<|channel|> + analysis/commentary/finalSeparate reasoning from answers
Tool Tokens<|call|>, <|constrain|>Enable agentic tool-use capabilities

For TinyStories (plain text): These special tokens aren't needed — Harmony tokenizes plain text identically to O200K base. The special tokens only matter when the training corpus contains structured conversations.

The code: tokenizer.py

import tiktoken

def get_tokenizer():
    o200k_base = tiktoken.get_encoding("o200k_base")
    tokenizer = tiktoken.Encoding(
        name="o200k_harmony",
        pat_str=o200k_base._pat_str,           # same BPE regex patterns
        mergeable_ranks=o200k_base._mergeable_ranks,  # same merge rules
        special_tokens={
            **o200k_base._special_tokens,       # inherit all base tokens
            "<|startoftext|>": 199998,
            "<|endoftext|>": 199999,
            "<|channel|>": 200005,
            "<|start|>": 200006,
            "<|end|>": 200007,
            "<|message|>": 200008,
            "<|call|>": 200012,
            ... reserved tokens up to 201088
        },
    )
    return tokenizer

Creating Input-Output Pairs

Why is this step special?

Unlike classification (image → label) or regression (features → price), language modeling has no explicit input-output split. The model creates its own targets by shifting the input right by one position. This makes it self-supervised (no labels needed) and auto-regressive (each output feeds back as input).

The code: data_loader.py

class TextDataset(Dataset):
    def __init__(self, tokens, max_length=8192, stride=8192):
        self.input_ids = []
        self.target_ids = []
        for i in range(0, len(tokens) - max_length, stride):
            input_chunk = tokens[i : i + max_length]
            target_chunk = tokens[i + 1 : i + max_length + 1]  # ← shifted right by 1
            self.input_ids.append(torch.tensor(input_chunk))
            self.target_ids.append(torch.tensor(target_chunk))
# stride = context_length ensures no overlap between sequences
# Each pair: input[i:i+4000] and target[i+1:i+4001]

Step 2: RMSNorm — Normalize Each Row

┌────────────────────────────────────────────────────────────────────┐
│ RMSNorm: normalize each row INDEPENDENTLY                         │
│ Formula: x_norm = x / RMS(x) × scale                              │
│ where RMS(x) = sqrt(mean(x²))                                     │
└────────────────────────────────────────────────────────────────────┘

Take row 0 of x: [0.1, 0.9, -0.4, 0.7]

1. Square each element:
   [0.01, 0.81, 0.16, 0.49]

2. Mean of squares:
   (0.01 + 0.81 + 0.16 + 0.49) / 4 = 1.47 / 4 = 0.3675

3. RMS = sqrt(0.3675) = 0.606

4. Divide each element by RMS:
   [0.1/0.606, 0.9/0.606, -0.4/0.606, 0.7/0.606]
 = [0.165, 1.485, -0.660, 1.155]

5. Multiply by learnable scale (assume scale = [1.0, 1.0, 1.0, 1.0]):
   Final row 0 = [0.165, 1.485, -0.660, 1.155]

(Same process for rows 1 and 2 — each normalized independently)

Scale parameter shape: (4,) = (hidden_size,) — one weight per dimension
Input shape:  (3, 4)
Output shape: (3, 4)  ← SAME! Normalization never changes shape.

Real model: (4096, 1024) → RMSNorm → (4096, 1024) — shapes never change

Token Embeddings

Why embeddings instead of one-hot vectors?

With a vocabulary of 201,088 tokens, a one-hot vector would be 201,088-dimensional and sparse (all zeros except one). Embeddings compress each token into a dense 2880-dimensional vector where similar words naturally cluster together after training. "day" and "night" end up near each other in vector space because they appear in similar contexts.

RMS Normalization

Why normalize at all?

Without normalization, activations can grow or shrink across layers, causing vanishing/exploding gradients. Normalization keeps values in a stable range, making training converge faster and smoother.

Why RMS Norm over Layer Norm?

Layer Norm computes both mean and variance, then centers and scales. RMS Norm skips the mean subtraction — it only divides by the root-mean-square. This is ~10-15% faster because it avoids the mean computation, and empirically performs just as well or better.

Layer Norm (GPT-2)

norm(x) = (x - mean(x)) / sqrt(var(x) + ε)
output = norm(x) * γ + β

Two steps: center (subtract mean), then scale (divide by std). Has both learnable scale (γ) and bias (β).

RMS Norm (GPT-OSS)

RMS(x) = sqrt(mean(x²) + ε)
output = (x / RMS(x)) * scale

One step: divide by RMS. Only has learnable scale — no bias. Faster, simpler.

The code

class RMSNorm(torch.nn.Module):
    def __init__(self, num_features, eps=1e-05, device=None):
        super().__init__()
        self.eps = eps
        self.scale = torch.nn.Parameter(
            torch.ones(num_features, device=device, dtype=torch.float32)
        )  # learnable scale, initialized to 1

    def forward(self, x):
        t = x.float()
        # Key: mean of SQUARES (not variance)
        t = t * torch.rsqrt(torch.mean(t**2, dim=-1, keepdim=True) + self.eps)
        return (t * self.scale).to(x.dtype)

# rsqrt = 1/sqrt — avoids computing sqrt then dividing
# No mean subtraction, no bias — just scale by 1/RMS

Attention: MHA → MQA → GQA

Multi-Head Attention (MHA) — the foundation

Evolution of attention mechanisms — GPT-OSS uses Grouped Query Attention (GQA) with 8:1 ratio

Why does the KV Cache matter?

During inference, for every new token generated, the model needs Key and Value matrices from all previous tokens. Without caching, these are recomputed from scratch — extremely wasteful.

KV Cache: Store K and V from previous steps, only compute the new token's K,V. But memory grows with (num_heads × head_dim × seq_len). GQA reduces num_KV_heads from 64 to 8, cutting KV cache memory by 8×.

Why not MQA (even smaller cache)?

MQA shares a single KV head across all 64 query heads. This destroys the ability to capture multiple perspectives simultaneously. GQA is the sweet spot — 8 groups of 8 queries, each group sharing one KV head.

How GQA is implemented in the code

Sliding Window Attention

Why limit how far back a token can look?

Computational cost: Full causal attention over 131K tokens = O(n²) = 17.1 billion operations. With sliding window of 128, it's O(n × w) = 16.8 million — a ~1000× reduction.

The tradeoff: Some very long-range dependencies might be missed. But with 24 transformer layers, information still propagates — layer 1's window covers tokens 1-128, layer 2 can attend to tokens that already saw 1-128, effectively reaching token 1 from token 256 through 2 layers. It's an indirect reach.

Full causal mask vs sliding window — the window "slides" diagonally, always width w

Rotary Positional Encodings (RoPE)

Why not just add position vectors?

Additive position embeddings (GPT-2 style) contaminate the semantic vector — a token at position 5 has a different vector than the same token at position 100, but the semantic meaning hasn't changed. RoPE solves this by rotating Q and K vectors. Rotation changes direction (encoding position) without changing magnitude (preserving meaning).

Why rotation works for relative position

When computing Q·K^T (attention score), rotation ensures the dot product naturally depends on the difference between positions, not absolute positions. Two tokens with the same relative distance always get the same attention contribution from position, regardless of where they appear in the sequence.

The rotation formula in code

def _apply_rotary_emb(x, cos, sin):
    x1, x2 = torch.chunk(x, 2, dim=-1)   # Split into pairs
    o1 = x1 * cos - x2 * sin               # Standard 2D rotation:
    o2 = x2 * cos + x1 * sin               # [cos θ, -sin θ] [x1]
    return torch.cat((o1, o2), dim=-1)       # [sin θ,  cos θ] [x2]

# The frequency for each pair:
freq = base ** (torch.arange(0, head_dim, 2) / head_dim)
# base = 150,000 for GPT-OSS (vs 10,000 in original RoPE)
# Higher base = slower decay = better long-context support

Frequency Pairs Explained: Why Different Speeds?

The Music Analogy

Think of position encoding like music with many instruments playing simultaneously:

  • High-frequency instruments (drums, hi-hat) — beat on every note. They tell you exactly where you are in the current bar. Small shifts are immediately noticeable.
  • Low-frequency instruments (bass drone, strings) — change slowly across the whole song. They tell you which section of the song you're in (intro, verse, chorus).

RoPE dimension pairs work the same way:

Low-index pairs (i = 0, 1, 2, ...) → HIGH frequency → FAST rotation

What they detect: Immediate word-order shifts. "The cat" vs "cat the" — these pairs rotate so fast that even a 1-position difference produces a noticeably different angle.

Example:

# Pair 0: θ₀ = 1/150000^(0/64) = 1.0
# At position 5:  angle = 5.0 radians (nearly full rotation!)
# At position 6:  angle = 6.0 radians
# Difference between adjacent positions = 1.0 radian = VERY noticeable
# → This pair can distinguish "dog bites man" from "man bites dog"

Real-world use: Syntax, grammar, adjacent word relationships. "The big dog" vs "The dog big" — pair 0 easily distinguishes these because adjacent tokens have very different rotation angles.

High-index pairs (i = 30, 31, 32) → LOW frequency → SLOW rotation

What they detect: Broad structural position. "Am I near the beginning or end of this paragraph?" These pairs barely change from one position to the next.

Example:

# At position 5:    angle = 0.0000425 radians (barely rotated)
# At position 6:    angle = 0.0000510 radians

# But at position 5 vs position 2000:
# angle_2000 = 0.017 radians — NOW there's a meaningful difference
# → This pair distinguishes "beginning of story" from "middle of story"

Real-world use: Long-range dependencies. Does the pronoun "he" in paragraph 3 refer to the character introduced in paragraph 1? High-index pairs encode this coarse positional info.

YaRN Scaling — Explained Simply

The Problem YaRN Solves

Suppose you trained the model on sequences up to 4096 tokens. During training, the model learned that:

  • Pair 0's angle at position 4096 = ~4096 radians (many full rotations)
  • Pair 31's angle at position 4096 = ~0.035 radians (barely turned)

Now at inference, someone gives you a 16384-token document (4× longer). What happens?

  • Pair 0's angle at position 16384 = ~16384 radians → Fine! More rotations is fine for high-freq (it wraps around).
  • Pair 31's angle at position 16384 = ~0.14 radians → Problem! The model NEVER saw angles this large for slow pairs during training!

Simple Example: The Clock Analogy

Imagine you trained someone to read time, but only up to 4 hours:

  • Second hand (high frequency): Spins many times in 4 hours. Already seen every possible position. Seeing it at hour 16 is fine — same positions, more laps.
  • Hour hand (low frequency): Only moved from 12 o'clock to 4 o'clock during training. At hour 16, it's at 4 o'clock position — BUT the person has never seen it travel through 5,6,7,...15 o'clock. Those are out-of-distribution angles!

YaRN's Solution: Treat High-Freq and Low-Freq Differently

YaRN says: "Don't apply the same scaling to all dimensions. Handle them based on their frequency."

For high-frequency pairs (fast rotation, low index):

These already cycle many times during training. Just extrapolate — let them keep spinning at the original rate. They've already seen all possible angles.

For low-frequency pairs (slow rotation, high index):

These would hit unseen angles. Interpolate — slow down the rotation so that position 16384 maps to the same angle range the model saw during training (0 to 4096).

For medium-frequency pairs:

Blend between interpolation and extrapolation using a smooth ramp. No hard boundary.

YaRN in code (simplified)

# scaling_factor = 4 (extending context 4× beyond training)

# For each dimension pair i:
interpolation = original_freq / scaling_factor   # Slow down rotation (fit in same range)
extrapolation = original_freq                     # Keep original speed

# Smooth ramp from 0 to 1 across dimensions:
# Low index (high freq): ramp ≈ 0  → uses extrapolation (keep spinning fast)
# High index (low freq): ramp ≈ 1  → uses interpolation (slow down)
# Middle: blended

mask = 1 - ramp
final_freq = interpolation * (1 - mask) + extrapolation * mask

Concrete numbers (with scaling_factor = 4)

# Pair 0 (high freq): original angle at pos 16384 = 16384.0 rad
#   → Extrapolate: keep 16384.0 rad (wraps around, model is fine with this)

# Pair 31 (low freq): original angle at pos 16384 = 0.14 rad
#   → Interpolate: 0.14 / 4 = 0.035 rad (same as pos 4096 during training!)
#   → Model stays in familiar territory

# Pair 15 (medium): blend of both
#   → 60% extrapolation + 40% interpolation

Result: Position 16384 produces attention patterns that "look" similar to what the model saw during training, even though the sequence is 4× longer. Quality degrades gracefully instead of catastrophically.

What are NTK alpha and beta?

ntk_alpha and ntk_beta control WHERE the transition from extrapolation to interpolation happens:

  • ntk_alpha = 1: The boundary for "fully extrapolated" dimensions (high freq end)
  • ntk_beta = 32: The boundary for "fully interpolated" dimensions (low freq end)
  • Between alpha and beta: smooth linear ramp blends the two

Think of it as: alpha says "above this frequency, don't touch." Beta says "below this frequency, fully compress." The ramp fills in the middle smoothly.

What is "concentration"?

concentration = 0.1 * log(scaling_factor) + 1.0
# For scaling_factor = 4: concentration = 0.1 * 1.386 + 1.0 = 1.139
# For scaling_factor = 1: concentration = 1.0 (no change)

When extending context, the attention scores get slightly more diffused (spread out). Concentration is a compensating factor that sharpens the cos/sin values slightly, keeping attention patterns crisp at extended lengths. It's a small correction (1.14× for 4× extension) but improves quality.

Attention Bias & Sinks

The Problem: Token-1 Obsession

Researchers discovered that in deeper transformer layers, every token pays massive attention to the first token, regardless of content. With sliding window attention, early tokens become inaccessible — so the excess attention gets dumped on random intermediate tokens, creating noisy, inflated attention scores.

The Solution: Learnable Bias Columns

Instead of designating actual tokens as sinks, GPT-OSS adds a learnable bias column to the attention scores matrix. This extra column absorbs the excess attention. After softmax, we drop the sink column — what remains are clean, undiluted scores for the real tokens.

The code

# AttentionBlock.__init__: one learnable sink per head
self.sinks = torch.nn.Parameter(
    torch.empty(num_attention_heads, dtype=torch.bfloat16)
)

# In sdpa():
S = S.reshape(n_heads, q_mult, 1, 1).expand(-1, -1, n_tokens, -1)
QK = torch.einsum("qhmd,khmd->hmqk", Q, K)  # Normal attention
QK *= sm_scale
QK += mask                                      # Causal + sliding mask
QK = torch.cat([QK, S], dim=-1)                # Append sink column
W = torch.softmax(QK, dim=-1)                  # Softmax over all including sink
W = W[..., :-1]                                # DROP the sink column
attn = torch.einsum("hmqk,khmd->qhmd", W, V)  # Attend only to real tokens

This is the core function that combines causal masking, sliding window, GQA broadcasting, attention sinks, and scaling — all in one compact function.

    # Q: (seq, 8 kv_groups, 8 q_per_group, 64 head_dim)
    # K: (seq, 8 kv_groups, 64 head_dim)
    # V: (seq, 8 kv_groups, 64 head_dim)

    # Broadcast K,V to match Q's grouped structure
    K = K[:, :, None, :].expand(-1, -1, q_mult, -1)
    V = V[:, :, None, :].expand(-1, -1, q_mult, -1)

    # Build mask: causal (upper triangle = -inf)
    mask = torch.triu(Q.new_full((n_tokens, n_tokens), -float("inf")), diagonal=1)

    # Add sliding window mask (lower triangle beyond window = -inf)
    if sliding_window > 0:
        mask += torch.tril(
            mask.new_full((n_tokens, n_tokens), -float("inf")),
            diagonal=-sliding_window
        )

    # Attention scores = Q · K^T, scaled
    QK = torch.einsum("qhmd,khmd->hmqk", Q, K) * sm_scale
    QK += mask[None, None, :, :]

    # Append sink column, softmax, then drop sink
    S = S.reshape(n_heads, q_mult, 1, 1).expand(-1, -1, n_tokens, -1)
    QK = torch.cat([QK, S], dim=-1)
    W = torch.softmax(QK, dim=-1)
    W = W[..., :-1]

    # Weighted sum of values
    attn = torch.einsum("hmqk,khmd->qhmd", W, V)
    return attn.reshape(n_tokens, -1)

GLU → GELU → SwiGLU: The Activation Function Evolution

Why does the activation function matter so much?

The activation function in the feed-forward network (FFN) determines what kind of non-linear transformations the model can learn. A better activation = better gradients during training = the model learns more effectively from the same data. This is a free performance boost — same parameter count, better results.

Step by step: Why each successor is better

ReLU → GLU: "Let the network decide what to pass through"

ReLU: max(0, x) — kills all negative values permanently. Simple but wastes half the neurons ("dead ReLU problem").

GLU (Gated Linear Unit): x ⊙ σ(Wx + b) — instead of hard-killing negatives, GLU uses a learned gate (sigmoid of a linear transform) that decides how much of each dimension to let through. The gate learns what information is useful.

GLU(x) = (x · W₁) ⊙ σ(x · W₂) where ⊙ = element-wise multiply

Why better: The sigmoid gate provides soft, learnable filtering. Information isn't killed — it's modulated.

GLU → GELU: "Use probability instead of a hard gate"

GELU (Gaussian Error Linear Unit): x · Φ(x) where Φ is the cumulative distribution function of a standard Gaussian. Instead of a learned sigmoid gate, GELU uses the probability that x would be greater than other inputs from a Gaussian distribution.

GELU(x) = x · Φ(x) ≈ 0.5x(1 + tanh[√(2/π)(x + 0.044715x³)])

Why better: GELU is smooth everywhere (good for gradients) and naturally gates based on magnitude — large positive values pass through, values near zero are partially blocked, and negative values are strongly dampened. Used in GPT-2, BERT.

GELU → Swish/SiLU: "Simpler formula, same idea"

Swish: x · σ(βx) where σ is sigmoid. When β=1, it's called SiLU (Sigmoid Linear Unit). Very similar curve to GELU but with a simpler, more computationally efficient formula.

Swish(x) = x · σ(αx) where α = 1.702 in GPT-OSS implementation

Key property: Non-monotonic — it dips slightly below zero for negative inputs before recovering. This small negative region helps the model learn richer representations than pure ReLU.

Swish → SwiGLU: "Combine Swish gating WITH linear gating"

SwiGLU combines two ideas: apply Swish activation to one path, keep a linear path (with +1 bias), and multiply them element-wise. The input is split into two halves — one gets the non-linear Swish, the other stays linear — and they gate each other.

SwiGLU(x) = Swish(x_glu) ⊙ (x_linear + 1)

Why +1 bias? Without it, if x_linear is near zero, the output is near zero regardless of the Swish path. Adding 1 ensures the linear gate starts at 1 (fully open) and adjusts from there — preventing vanishing signals.

Why two paths? The Swish path provides non-linearity and feature detection. The linear path provides direct information flow. Their product lets the network selectively amplify or suppress features based on both non-linear and linear views.

GPT-OSS SwiGLU implementation

def swiglu(x, alpha=1.702, limit=7.0):
    # Split input into interleaved glu/linear channels
    x_glu, x_linear = x[..., ::2], x[..., 1::2]  # even/odd indices

    # Clamp for numerical stability (prevents overflow in sigmoid)
    x_glu = x_glu.clamp(max=limit)        # cap the Swish input
    x_linear = x_linear.clamp(-limit, limit)

    # Swish activation: x * sigmoid(alpha * x)
    out_glu = x_glu * torch.sigmoid(alpha * x_glu)

    # Gate: Swish output * (linear + 1)
    return out_glu * (x_linear + 1)  # +1 bias prevents vanishing

Mixture of Experts & The Router

Why replace one big FFN with many small ones?

Capacity without cost: A single FFN with 4× expansion processes every token through all parameters. With MoE, you have 32 experts but only 4 are active per token — giving you 32× the total capacity while using only 4/32 = 12.5% of the compute per forward pass.

Specialization: Different experts learn to handle different types of content. One expert might specialize in verbs, another in spatial descriptions, another in emotions. The router learns which expert handles which type of input best.

Only K=4 experts are active per token — the rest are completely dormant (no compute)

The Router: How experts are selected

The router is simply a linear layer that maps each token's hidden state (2880-d) to a score for each expert (32-d). Then torch.topk selects the top K=4 experts. Their scores are softmaxed to become blend weights.

# Router: single linear projection → scores for each expert
self.gate = nn.Linear(hidden_size, num_experts)  # 2880 → 32

def forward(self, x):
    t = self.norm(x)                              # RMS normalize first
    g = self.gate(t)                              # (seq, 32) router scores

    # Select top-K experts with highest scores
    experts = torch.topk(g, k=self.experts_per_token, dim=-1)
    expert_weights = F.softmax(experts.values, dim=-1)  # normalize K scores
    expert_indices = experts.indices                     # which K experts

    output = torch.zeros_like(t)
    for expert_idx in range(self.num_experts):
        mask = (expert_indices == expert_idx).any(dim=-1)
        if not mask.any():
            continue  # Skip dormant experts entirely — zero compute!

        # Only process tokens assigned to this expert
        expert_input = t[mask]
        expert_out = self.experts[expert_idx][0](expert_input)  # Linear 1
        expert_out = swiglu(expert_out)                          # SwiGLU
        expert_out = self.experts[expert_idx][1](expert_out)  # Linear 2

        output[mask] += expert_out * weights.unsqueeze(-1)  # Weighted add

    return x + output  # Residual connection

What this file does

Creates the tokenizer that converts text ↔ numbers. The model never sees raw text — it only works with integer IDs. This file builds the mapping between the two.

import tiktoken
# tiktoken is OpenAI's BPE tokenizer library.
# BPE (Byte Pair Encoding) is an algorithm that finds the most common
# character sequences in text and assigns them an ID number.
# "learning" might become ["learn", "ing"] → [45672, 278]

def get_tokenizer():
    # STEP 1: Load OpenAI's pre-built tokenizer with ~200,000 merge rules.
    # These rules define how to split text into subword tokens.
    # Example merges: 'h'+'e' → 'he', 'he'+'ll' → 'hell', 'hell'+'o' → 'hello'
    o200k_base = tiktoken.get_encoding("o200k_base")

    # STEP 2: Create our custom "Harmony" variant.
    # We keep ALL the original BPE rules (same text splitting behavior)
    # but ADD special tokens for chat/tool/reasoning features.
    tokenizer = tiktoken.Encoding(
        name="o200k_harmony",

        # pat_str: The regex pattern that splits text BEFORE BPE applies.
        # It defines word boundaries: spaces, punctuation, contractions, etc.
        # Example: "don't go" → ["don", "'t", " go"] → then BPE on each
        pat_str=o200k_base._pat_str,

        # mergeable_ranks: The actual BPE merge table.
        # This is a dict of {byte_sequence: priority_rank}.
        # Higher rank = merged later. The tokenizer applies merges in order.
        # We inherit ALL ~200K merges from O200K — same tokenization as GPT-4.
        mergeable_ranks=o200k_base._mergeable_ranks,

        # special_tokens: Tokens with FIXED IDs that are never split by BPE.
        # These are used for formatting, not for natural language content.
        special_tokens={
            # Inherit all base special tokens (like <|endoftext|>)
            **o200k_base._special_tokens,

            # === Document boundaries ===
            "<|startoftext|>": 199998,  # Marks beginning of a document
            "<|endoftext|>": 199999,    # Marks end of a document

            # === Tool calling tokens ===
            "<|return|>": 200002,       # Marks where a function returns its result
            "<|constrain|>": 200003,    # Constrains output to specific format/schema
            "<|call|>": 200012,         # Marks a tool/function call

            # === Chat/reasoning structure ===
            "<|channel|>": 200005,      # Separates reasoning channels
                                        # (analysis / commentary / final answer)
            "<|start|>": 200006,        # Start of a structured block
            "<|end|>": 200007,          # End of a structured block
            "<|message|>": 200008,      # Separates messages in a conversation
        } | {
            # Reserve 1075 more IDs (200013–201087) for future features.
            # These aren't used yet, but they ensure the vocab_size stays
            # fixed even as new features are added later.
            f"<|reserved_{i}|>": i for i in range(200013, 201088)
        },
    )
    return tokenizer

# USAGE:
# tokenizer = get_tokenizer()
# tokenizer.encode("Hello world") → [9906, 1917]
# tokenizer.decode([9906, 1917]) → "Hello world"
# tokenizer.n_vocab → 201088

  • pat_str = how to split text into pre-BPE chunks (regex)
  • mergeable_ranks = the BPE merge rules (which byte sequences become tokens)
  • special_tokens = fixed-ID tokens for structure (chat, tools, etc.)
  • For TinyStories training, only the BPE rules matter — special tokens aren't in the data
  • Total vocab = 201,088 (200K BPE + ~1K special/reserved)

# This is WHERE the magic happens — the self-supervised objective.
# ═══════════════════════════════════════════════════════════════
class TextDataset(Dataset):
    # PyTorch Dataset that slices the token stream into fixed-length windows.
    #
    # Visualization (context_len=6 for simplicity):
    #
    # All tokens: [The, cat, sat, on, the, mat, A, dog, ran, to, the, park, ...]
    #
    # Window 1:  input  = [The, cat, sat, on,  the, mat]  (positions 0-5)
    #            target = [cat, sat, on,  the, mat, A  ]  (positions 1-6)
    #
    # Window 2:  input  = [A,   dog, ran, to, the, park]  (positions 6-11)
    #            target = [dog, ran, to, the, park, ...]  (positions 7-12)
    #
    # Notice: target = input shifted right by 1 position!
    # This forces the model to predict what comes NEXT at every position.

    def __init__(self, tokens, max_length=8192, stride=8192):
        self.input_ids, self.target_ids = [], []

        # Slide a window across the token stream.
        # stride = max_length means NO OVERLAP between windows.
        # (stride < max_length would create overlapping windows,
        #  which wastes compute on redundant data)
        for i in range(0, len(tokens) - max_length, stride):
            # Input: tokens at positions [i, i+1, ..., i+max_length-1]
            input_chunk = tokens[i : i + max_length]

            # Target: tokens at positions [i+1, i+2, ..., i+max_length]
            # This is the input shifted right by 1 — the "answer key"
            target_chunk = tokens[i+1 : i + max_length + 1]

            # Store as PyTorch tensors (GPU-ready number arrays)
            self.input_ids.append(torch.tensor(input_chunk))
            self.target_ids.append(torch.tensor(target_chunk))

    def __len__(self):
        return len(self.input_ids)  # Number of training sequences

    def __getitem__(self, idx):
        # Return one (input, target) pair. PyTorch calls this to build batches.
        return self.input_ids[idx], self.target_ids[idx]

val_dataset = TextDataset(val_tokens, max_length=context_len, stride=context_len)

    train_dataset,
    batch_size=batch_size,   # Group 5 sequences together per GPU step
    shuffle=True,            # Randomize order each epoch (prevents memorization)
    num_workers=8,           # 8 CPU threads load data in parallel while GPU trains
    pin_memory=True          # Pre-allocate in GPU-friendly memory (faster transfer)
)
val_loader = DataLoader(
    val_dataset,
    batch_size=batch_size,
    shuffle=False,           # Don't shuffle validation — we want reproducible eval
    num_workers=8,
    pin_memory=True
)

# Free raw text from memory — we only need the DataLoaders now
del dataset, train_text, val_text
gc.collect()  # Force Python garbage collector to reclaim freed memory

TransformerBlock — one layer of the "brain"

Each transformer block does TWO things: (1) look at other tokens via attention, (2) process the result via expert FFNs. Both have residual connections.

class TransformerBlock(torch.nn.Module):
    # One "layer" of the transformer. The model stacks 24 of these.
    #
    # Data flow through one block:
    #   input (2880-d vector per token)
    #     → AttentionBlock: "look at other tokens, gather info"
    #     → MLPBlock: "process gathered info through expert FFNs"
    #   output (2880-d vector per token, same shape as input!)
    #
    # The residual connections inside each sub-block ensure that
    # information flows smoothly through all 24 layers without
    # vanishing gradients.

    def __init__(self, config, layer_idx, device):
        super().__init__()
        # layer_idx determines: even=sliding window, odd=full causal
        self.attn = AttentionBlock(config, layer_idx, device)
        self.mlp = MLPBlock(config, device)

    def forward(self, x):
        # x shape: (sequence_length, 2880)

        x = self.attn(x)   # Internally does: x = x + attention(norm(x))
                            # After this: each token's vector now contains info
                            # it "pulled" from other tokens via attention.

        x = self.mlp(x)    # Internally does: x = x + moe(norm(x))
                            # After this: expert FFNs have transformed the info
                            # (computing, reasoning, pattern matching).

        return x           # Same shape: (sequence_length, 2880)

Transformer — the complete model (forward pass)

The full model is surprisingly simple: embed → repeat transformer block 24× → project to vocabulary. All the complexity is inside the blocks.

class Transformer(torch.nn.Module):
    def __init__(self, config, device):
        super().__init__()

        # EMBEDDING TABLE: converts token IDs → dense vectors.
        # This is a lookup table with 201,088 rows × 2880 columns.
        # Token ID 5674 → row 5674 of this table → a 2880-d vector.
        # These vectors are LEARNED during training. Initially random.
        self.embedding = nn.Embedding(config.vocab_size, config.hidden_size)
        # Memory: 201088 × 2880 × 2 bytes (bf16) ≈ 1.1 GB

        # TRANSFORMER BLOCKS: the model's "thinking" layers.
        # Stacked 24 deep. Each block does attention → MoE.
        # Alternating pattern:
        #   Block 0: sliding window (local, fast)
        #   Block 1: full causal (global, slow)
        #   Block 2: sliding window
        #   Block 3: full causal
        #   ... and so on
        self.block = nn.ModuleList([
            TransformerBlock(config, layer_idx, device)
            for layer_idx in range(config.num_hidden_layers)
        ])

        # FINAL NORM: stabilize activations before the output projection.
        # Without this, the last block's output could have wildly varying
        # magnitudes, making the logits noisy.
        self.norm = RMSNorm(config.hidden_size)

        # UNEMBEDDING: project from hidden space → vocabulary space.
        # This is the "reverse" of embedding: vectors → word scores.
        # Output: one score (logit) for EVERY token in the vocabulary.
        # No bias term — the raw logit values are what we need.
        self.unembedding = nn.Linear(config.hidden_size, config.vocab_size, bias=False)
        # Memory: 2880 × 201088 × 2 bytes ≈ 1.1 GB

    def forward(self, x):
        # x: tensor of token IDs, shape (sequence_length,)
        # Example: x = [12805, 5765, 264, 892, ...] (4096 token IDs)

        x = self.embedding(x)       # (seq,) → (seq, 2880)
                                     # Each token ID is now a 2880-d vector.
                                     # Initially these are random; training makes them meaningful.

        for block in self.block:     # 24 transformer blocks, applied sequentially
            x = block(x)             # (seq, 2880) → (seq, 2880)
                                     # Each block refines the representation:
                                     # Early blocks: learn surface patterns (word order, grammar)
                                     # Middle blocks: learn semantics (meaning, relationships)
                                     # Late blocks: learn task-specific patterns (story structure)

        x = self.norm(x)             # Final normalization (stabilize magnitudes)

        x = self.unembedding(x)      # (seq, 2880) → (seq, 201088)
                                     # Now each position has 201,088 scores — one per vocab token.
                                     # The highest score = model's prediction for next token.

        return x                    # These are "logits" (raw scores, not probabilities yet).
                                     # Training: cross-entropy(logits, true_next_token)
                                     # Inference: softmax(logits) → sample → next token

Unembed:  (4096, 201088)       # project to vocabulary scores

Every intermediate step is (seq_len, 2880). The shape ONLY changes at the very start (embed) and very end (unembed). This uniformity is what makes transformers so elegant.

Code Walkthrough: trainer.py

What this file does

Implements the training loop — the iterative process where the model gets better. Each step: show the model some text, compare its predictions to reality, compute how wrong it was, and nudge all weights slightly to be less wrong next time.

Loss function: how wrong is the model?

Cross-entropy loss measures: "how surprised was the model by the correct answer?" If the model assigns high probability to the true next token → low loss (good). If it assigns low probability → high loss (bad). We want to minimize this.

def calcc(input_batch, target_batch, model, device):
    # This function computes the average loss over a batch of sequences.
    # It answers: "on average, how surprised was the model by the correct token?"

    total_loss = 0

    # Process each sequence in the batch one at a time
    # (because our model processes single sequences, not batched)
    for i in range(len(input_batch)):
        # Move input tokens to GPU
        inp = input_batch[i].to(device)   # shape: (4000,) — token IDs

        # FORWARD PASS: model predicts next token at every position
        logits = model(inp)                # shape: (4000, 201088)
                                            # logits[i][j] = score for token j being
                                            # the next token after position i

        # Move target tokens (the correct answers) to GPU
        tgt = target_batch[i].to(device)   # shape: (4000,) — the TRUE next tokens

        # CROSS-ENTROPY: compare predictions vs reality
        # For each of the 4000 positions:
        #   1. Take the 201,088 logits (scores for each vocab token)
        #   2. Apply softmax → probabilities
        #   3. Take -log(probability of the TRUE token)
        #   4. Average across all positions
        # Example: if true token has probability 0.8 → loss = -log(0.8) = 0.22 (low, good!)
        #          if true token has probability 0.01 → loss = -log(0.01) = 4.6 (high, bad!)
        loss = F.cross_entropy(logits, tgt)

        total_loss += loss

    # Return average loss across the batch
    return total_loss / len(input_batch)
    # Typical starting loss: ~12.2 (random model, log(201088) ≈ 12.2)
    # Good loss after training: ~1.7 (model predicts well)

Training loop: the actual learning process

Training = repeating this cycle millions of times:

  1. Forward: Feed tokens → get predictions
  2. Loss: How wrong were the predictions?
  3. Backward: Compute gradient (which direction should each weight move?)
  4. Update: Nudge all weights slightly in the right direction
  5. Schedule: Adjust how big the nudges are
def trainer(model, train_loader, val_loader, device):
    # ═══════════════════════════════════════════════════════════
    # HYPERPARAMETERS: control how fast and how well the model learns
    # ═══════════════════════════════════════════════════════════
    learning_rate = 3e-4   # "Step size" — how much to change weights per step.
                            # Too high → overshoots, loss goes up. Too low → learns too slowly.
                            # 3e-4 (0.0003) is the standard starting point for LLM training.

    warmup_steps = 100     # Steps where LR ramps up from 0.1× to 1× of max.
                            # Why? At start, weights are RANDOM → gradients are HUGE and NOISY.
                            # Large LR + huge gradients = catastrophe. Small LR first = safe start.

    min_lr = 3e-5          # Minimum LR at the end of training.
                            # Never fully stop learning, but make very small adjustments.

    # ═══════════════════════════════════════════════════════════
    # OPTIMIZER: AdamW — the algorithm that updates weights
    # ═══════════════════════════════════════════════════════════
    # AdamW = Adam (adaptive learning rate per parameter) + Weight Decay
    # Why Adam? SGD uses the same step size for all params.
    # Adam tracks a RUNNING AVERAGE of gradients (momentum) and their
    # VARIANCE. Params with consistent gradients get larger steps.
    # Params with noisy gradients get smaller steps. Smart and adaptive!
    optimizer = torch.optim.AdamW(
        model.parameters(),
        lr=learning_rate,
        weight_decay=0.1    # L2 regularization: shrink weights toward 0.
                             # Prevents overfitting by penalizing large weights.
                             # 0.1 is aggressive (standard is 0.01) but works for LLMs.
    )

    # ═══════════════════════════════════════════════════════════
    # LEARNING RATE SCHEDULE: how LR changes during training
    # ═══════════════════════════════════════════════════════════
    # Phase 1: Warmup (steps 0→100)
    # LR ramps from 3e-5 → 3e-4 linearly.
    # "Let the model orient itself before running."
    scheduler_warmup = LinearLR(
        optimizer,
        start_factor=0.1,    # Start at 10% of max LR = 3e-5
        end_factor=1.0,      # Ramp to 100% of max LR = 3e-4
        total_iters=warmup_steps
    )

    # Phase 2: Cosine decay (steps 100→end)
    # LR follows cos(x) from 3e-4 down to 3e-5.
    # "Gradually reduce step size as we approach optimal weights."
    # Why cosine? Stays high longer than linear (more exploration time),
    # then drops quickly at the end (precise convergence).
    scheduler_decay = CosineAnnealingLR(
        optimizer,
        T_max=max_iters - warmup_steps,  # Duration of cosine phase
        eta_min=min_lr                    # Floor: never go below 3e-5
    )

    # Combine: warmup first, then cosine. Switch at milestone.
    scheduler = SequentialLR(
        optimizer,
        schedulers=[scheduler_warmup, scheduler_decay],
        milestones=[warmup_steps]  # Switch from warmup → cosine at step 100
    )

    # ═══════════════════════════════════════════════════════════
    # THE ACTUAL TRAINING LOOP
    # ═══════════════════════════════════════════════════════════
    for epoch in range(num_epochs):
        for input_batch, target_batch in train_loader:

            # 1. FORWARD PASS: run data through model, get predictions
            loss = calcc(input_batch, target_batch, model, device)
            #    loss = single number saying "how wrong was the model"

            # 2. BACKWARD PASS: compute gradients via backpropagation
            #    For EVERY weight in the model (millions of them),
            #    compute: "if I increase this weight slightly, does loss go up or down?"
            #    This gradient tells us the DIRECTION to move each weight.
            loss.backward()

            # 3. GRADIENT CLIPPING: safety check
            #    Sometimes gradients are HUGE (especially in bf16).
            #    Cap them at magnitude 1.0 to prevent catastrophic updates.
            torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)

            # 4. OPTIMIZER STEP: actually update the weights
            #    Each weight is nudged: weight -= lr * gradient
            #    (AdamW also applies momentum, variance scaling, and weight decay)
            optimizer.step()

            # 5. ZERO GRADIENTS: clear for next iteration
            #    Without this, gradients would ACCUMULATE across steps
            #    (sometimes intentional for gradient accumulation, but not here)
            optimizer.zero_grad(set_to_none=True)
            #    set_to_none=True is slightly faster than setting to 0

            # 6. UPDATE LEARNING RATE: follow the warmup → cosine schedule
            scheduler.step()

            # 7. EVALUATE periodically (every eval_freq steps)
            #    Run model on VALIDATION data (never seen during training)
            #    to check if the model generalizes or is just memorizing

            # 8. SAVE BEST MODEL
            #    If val_loss is the lowest we've ever seen, save weights.
            #    This is our "insurance" — even if training later goes bad,
            #    we have the best version saved.

Why this specific setup?

  • AdamW over SGD: Adaptive per-parameter LR. Embeddings need different step sizes than attention weights. Adam handles this automatically.
  • Weight decay 0.1: Aggressively prevents overfitting. On small datasets like TinyStories, the model can memorize stories without generalizing. Weight decay forces it to find simpler, more general patterns.
  • Gradient clipping 1.0: bf16 has limited range (max ~65504). Large gradients can overflow. Clipping at 1.0 ensures numerical stability.
  • Cosine > Linear decay: Spends more time at high LR (exploring), then quickly converges at the end. Empirically gives ~5-10% better final loss than linear.
  • Warmup 100 steps: Random weights → huge variance in gradients. First 100 steps with low LR = let gradients stabilize before accelerating.

Code Walkthrough: inference.py

What this file does

Generates text one token at a time. The model was trained to predict the next token — inference simply USES that ability repeatedly: predict next → append → predict next → append → ...

def generate_text(model, prompt, max_tokens=100, temperature=0.8, top_k=50):
    # STEP 1: Convert the user's text prompt into token IDs.
    # "Once upon a time" → [12805, 5765, 264, 892]
    idx = text_to_token_ids(prompt, tokenizer).to(device)
    # idx is now a 1-D tensor of token IDs on the GPU

    # STEP 2: Auto-regressive generation loop.
    # Repeat max_tokens times: predict one token, append it, repeat.
    for _ in range(max_tokens):

        # Keep only the last `context_len` tokens.
        # The model has a fixed-size window (e.g., 4096 tokens).
        # If the sequence grows beyond that, we must truncate.
        # (In production, KV-cache would avoid recomputing old tokens)
        idx_cond = idx[-context_len:]

        # FORWARD PASS (no gradient computation — we're not training!)
        # inference_mode() is faster + uses less memory than no_grad()
        with torch.inference_mode():
            logits = model(idx_cond)
            # logits shape: (seq_len, 201088)
            # Each position predicts the NEXT token.
            # We only care about the LAST position's prediction
            # (what comes after the entire prompt so far?)

        # Take ONLY the last position's logits
        logits = logits[-1, :]    # shape: (201088,) — one score per vocab token

        # ════════════════════════════════════════════════════════
        # TEMPERATURE SCALING: controls randomness of output
        # ════════════════════════════════════════════════════════
        # Dividing logits by temperature BEFORE softmax changes the distribution:
        #
        #   temperature = 0.3: logits become LARGER → softmax is SHARPER
        #     → probabilities: [0.95, 0.03, 0.01, 0.01, ...]
        #     → almost always picks the top token (safe, repetitive)
        #
        #   temperature = 1.0: logits unchanged → original distribution
        #     → probabilities: [0.4, 0.2, 0.15, 0.1, ...]
        #     → reasonable variety
        #
        #   temperature = 1.5: logits become SMALLER → softmax is FLATTER
        #     → probabilities: [0.2, 0.18, 0.16, 0.14, ...]
        #     → highly random, creative but sometimes incoherent
        #
        logits = logits / temperature

        # ════════════════════════════════════════════════════════
        # TOP-K FILTERING: remove unlikely candidates
        # ════════════════════════════════════════════════════════
        # Problem: even after temperature, there are 201,088 options.
        # Most have tiny probability but CAN be sampled (like "xylophone"
        # after "The cat sat on the"). Top-K prevents this.
        #
        # Strategy: find the 50th-highest score. Set everything below it to -∞.
        # After softmax, -∞ → probability 0. Only top 50 candidates remain.
        if top_k is not None:
            v, _ = torch.topk(logits, min(top_k, logits.size(-1)))
            # v[-1] is the 50th highest score (the "cutoff")
            logits[logits < v[[-1]]] = -float('Inf')
            # Now: 50 tokens have real scores, 201,038 tokens have -∞

        # ════════════════════════════════════════════════════════
        # SAMPLING: pick one token from the distribution
        # ════════════════════════════════════════════════════════
        # Convert logits → probabilities (sum to 1.0)
        probs = F.softmax(logits, dim=-1)

        # MULTINOMIAL SAMPLING: randomly draw one token, weighted by probability.
        # If probs = [0.7, 0.2, 0.1], it picks token 0 ~70% of the time.
        # This is what makes generation non-deterministic (different each run!).
        idx_next = torch.multinomial(probs, 1)   # shape: (1,) — one token ID

        # APPEND the new token to our growing sequence
        idx = torch.cat((idx, idx_next), dim=0)
        # Next iteration: the model sees this new token too!
        # It becomes part of the context for generating the NEXT token.

    # STEP 3: Convert token IDs back to text
    # [12805, 5765, 264, 892, 8234, ...] → "Once upon a time there was..."
    return token_ids_to_text(idx, tokenizer)

Deep Question: Why Different Q if K,V are Shared?

The Question

In GQA, multiple query heads share the same K and V. If they all see the same "memory", won't they all produce the same output? Why keep multiple Q heads at all? Why not just one?

The Core Insight: Q = "What I'm looking for", K,V = "What exists"

Attention is fundamentally asymmetric:

  • K, V = the library (shared knowledge base — what information exists)
  • Q = the reader (intent — what specific information am I looking for right now?)

You can share the library. You cannot share the intent.

Mathematical Proof

Setup: Shared K, V — Different Q heads

# Shared memory (same for all heads in the group)
K = [[1, 0],    # Token 1's key: "I am about syntax"
     [0, 1]]    # Token 2's key: "I am about semantics"

V = [[10, 0],   # Token 1's value: syntax information
     [0, 20]]   # Token 2's value: semantic information

# Two different query heads (different learned projections)
Q₁ = [1, 0]    # Head 1 asks: "Give me syntax info"
Q₂ = [0, 1]    # Head 2 asks: "Give me semantic info"

Computation: Same K,V → Different Outputs!

# Head 1: Q₁ · K^T = [1,0] · [[1,0],[0,1]]^T = [1, 0]
# → Attention weights after softmax: [~1.0, ~0.0] (attends to token 1)
# → Output: [1.0, 0.0] · V = [10, 0]  ← syntax info!

Head 1 output = [10, 0]   # Got syntax

# Head 2: Q₂ · K^T = [0,1] · [[1,0],[0,1]]^T = [0, 1]
# → Attention weights after softmax: [~0.0, ~1.0] (attends to token 2)
# → Output: [0.0, 1.0] · V = [0, 20]  ← semantic info!

Head 2 output = [0, 20]   # Got semantics

Same K, same V. Different Q → completely different outputs!

The Library Analogy

Different Q heads extract different information from the same shared K,V memory

Why NOT just one head then?

If you had only 1 Q head:

  • Only one type of question can be asked at each position
  • Only one attention pattern per layer
  • Cannot simultaneously track: syntax + semantics + entities + sentiment

Language needs multiple simultaneous perspectives at every position. "The cat sat on the mat" at position 5 needs to simultaneously know: what's the subject (cat), what's the action (sat), what's the location (mat), is this a statement or question.

The Real Answer: Attention is Asymmetric

K, V = "what exists" (memory) → can be shared

Q = "what I want" (intent) → must be diverse

The projection Q_i = x · W_Q_i means each head learns a different transformation of the same input. Even though they query the same K,V store, they extract completely different information.

GQA Insight: The memory (K,V) only needs moderate diversity — 4 KV heads capture enough. But the questions asked (Q) need maximum diversity — 16 heads asking different things about that memory.

What does GQA actually sacrifice?

AspectFull MHA (16Q : 16KV)GQA (16Q : 4KV)Impact
Q diversity16 unique projections16 unique projectionsNo loss — queries are all different
K diversity16 unique key spaces4 unique key spacesSlight loss — fewer "indexing schemes"
V diversity16 unique value spaces4 unique value spacesSlight loss — fewer "answer formats"

Empirical finding: K,V diversity beyond 4-8 heads gives diminishing returns. Most of the model's expressiveness comes from Q diversity. This is why GQA works — it keeps what matters (Q) and shares what can be shared (KV).

Deep Question: Why Rotate Instead of Add for Position?

The Question

GPT-2 adds a position vector to the token embedding: x = token_emb + pos_emb. RoPE rotates Q and K instead. Why is rotation better? What goes wrong with addition?

Problem 1: Semantic Pollution

Additive position (GPT-2)

# "king" at position 5 vs position 500:
emb_pos5   = king_vector + pos_5_vector     # magnitude changes!
emb_pos500 = king_vector + pos_500_vector   # different magnitude!

# The MEANING of "king" is now entangled with WHERE it appears
# ||emb_pos5|| ≠ ||emb_pos500|| even though semantics are identical

Addition changes the vector's magnitude — "polluting" semantic meaning with positional noise.

Rotational position (RoPE)

# "king" at position 5 vs position 500:
q_pos5   = rotate(king_q_vector, angle=θ*5)
q_pos500 = rotate(king_q_vector, angle=θ*500)

# ||q_pos5|| == ||q_pos500|| — magnitude preserved!
# Only the DIRECTION changes, not the semantic content

Rotation preserves magnitude: ||rotated(v)|| = ||v||. Semantic meaning stays intact.

Problem 2: Relative Position Doesn't Emerge Naturally

Why relative position matters

"The cat" should produce similar attention whether it appears at position (5,6) or (100,101). What matters is the distance between tokens, not absolute positions.

With addition: (token_a + pos_5) · (token_b + pos_6)^T — this dot product mixes token-token, token-position, and position-position terms in a complex, non-decomposable way.

With rotation: rotate(Q, pos_a) · rotate(K, pos_b)^T = f(Q, K, pos_a - pos_b) — the dot product ONLY depends on the relative distance (pos_a - pos_b), not absolute positions.

Problem 3: Fixed Length

Additive: Position embeddings are a learned lookup table of size (max_seq_len, hidden_dim). If you train with max_seq_len=1024, you literally cannot use position 1025 — there's no learned vector for it.

RoPE: Rotation angles are computed analytically from a formula: θ_i * position. Any position is valid — no learned table, no maximum. YaRN further extends this by blending interpolation/extrapolation for positions beyond training.

The Frequency Hierarchy

# RoPE frequency for dimension pair i:
θ_i = 1 / (base ^ (2i / d))

# Low i (early dimensions):  high frequency → changes fast with position
#   → captures FINE word-order shifts ("the cat" vs "cat the")

# High i (later dimensions): low frequency → changes slowly
#   → captures BROAD structural patterns (paragraph position, story arc)

This is analogous to Fourier analysis: different frequencies capture different scales of pattern. The model naturally learns which dimension-pairs to attend to based on what distance matters.

Deep Question: Why Remove the Mean Subtraction?

The Question

LayerNorm does two things: (1) subtract mean, (2) divide by std. RMSNorm only does the division (by RMS). Why is the mean subtraction unnecessary? Doesn't centering help?

What does each operation do?

Mean subtraction (centering)

Removes the "DC component" — the constant offset of the vector. Centers everything around zero.

Hypothetical benefit: Removes bias from activations, helps gradient flow.

In practice: With residual connections, the network can easily learn any bias it needs. The centering doesn't provide unique value.

Division by RMS (magnitude normalization)

Brings all vectors to approximately unit scale regardless of their absolute magnitude.

Why this matters:

  • Prevents some tokens from having much larger activations than others
  • Ensures the learnable scale parameter operates in a consistent range
  • Prevents exploding/vanishing signals through deep layers

This is the part that actually matters for training stability.

The Mathematical Argument

# LayerNorm: (x - mean(x)) / std(x) * γ + β
# Parameters: γ (scale) + β (bias) = 2N parameters

# RMSNorm: x / RMS(x) * scale
# Parameters: scale only = N parameters

# Key insight: the bias β in LayerNorm can learn to add back any mean
# So subtracting mean + adding learnable bias = net zero benefit
# The residual connection also provides a "free" bias pathway

Speed Difference

Subtract mean1 elementwise op—
Compute variance1 reduction + square—
Compute RMS (mean of squares)—1 reduction + square
Divide1 elementwise op1 elementwise op

Total~6 ops~3 ops

RMSNorm is ~15% faster end-to-end because normalization runs at every layer × every token × every training step. Small per-call savings compound massively.

Deep Question: Why Gate the Activation Function?

The Question

GELU already works well (GPT-2 uses it). Why add gating (SwiGLU)? What does the "gate" actually provide? And why the +1 bias on the linear path?

The Problem with Single-Path Activations

# ReLU:  max(0, x)   — binary: either pass or kill
# GELU:  x · Φ(x)   — smooth: probability of passing

# Both are single-path: one transformation, one decision
# The SAME neurons decide both WHAT to compute and WHETHER to pass it

This couples two distinct operations into one function. The network must use the same parameters to decide both the content and the gating.

What Gating Provides: Separation of Concerns

# SwiGLU: Two separate paths with different roles

gate_path   = input[even_indices]     # "SHOULD this pass?"
linear_path = input[odd_indices]      # "WHAT should pass?"

gate = swish(gate_path)               # Decision: 0 to ~gate_path
output = gate * (linear_path + 1)    # Gate controls linear flow

Key insight: The network now has separate parameters for:

  • Content: What information to represent (linear path)
  • Selection: Which features to amplify or suppress (gate path)

This is like having a data bus (linear) with a control signal (gate). Much more expressive than a single wire.

Why +1 Bias?

# Without +1:
output = gate * linear_path
# If linear_path ≈ 0, output ≈ 0 regardless of gate
# The gate becomes useless — can't rescue a zero signal

# With +1:
output = gate * (linear_path + 1)
# Default state: gate * 1 = gate (the gate itself flows through)
# linear_path adjusts UP or DOWN from this baseline
# The gate always has something meaningful to work with

Think of it as: the system starts "open" (passing the gate signal) and the linear path modulates from there. Without +1, the system starts "closed" and must actively open — harder to learn from random initialization.

Why SwiGLU beats GELU: Gradient Analysis

GELU gradient: d/dx[x·Φ(x)] = Φ(x) + x·φ(x) — can be very small for large negative x, leading to slow learning.

SwiGLU gradient through linear path: The linear path has a direct gradient connection (always ≠ 0). Even if the gate is partially closed, gradients flow through the linear path to the input. This means:

  • No "dead neuron" problem (unlike ReLU)
  • No vanishing gradients for negative inputs (unlike GELU)
  • The gate provides multiplicative modulation without blocking gradient flow

Deep Question: Why Not One Big FFN?

The Question

If we want more capacity, why not just make one big FFN larger? Why split into multiple experts with a complex routing mechanism? What does specialization actually buy?

The Scaling Problem

# Option A: One big FFN
# Double capacity → double the intermediate dimension
# Linear(1024 → 4096) → GELU → Linear(4096 → 1024)
# Compute per token: O(d × 4d) = O(4d²)
# EVERY token pays the full cost, even simple ones like "the"

# Option B: 4 small experts, top-2 routing
# Linear(1024 → 2048) → SwiGLU → Linear(1024 → 1024) × 4 experts
# Compute per token: O(2 × d × 2d) = O(4d²) — same! But:
# - 4× total parameter capacity
# - Each expert can specialize
# - Full GPT-OSS: 32 experts, 4 active = 8× capacity, same compute

Why Specialization Happens Naturally

The router is a simple Linear layer that learns during training. Over time:

  • Expert 0 might specialize in function words ("the", "is", "a")
  • Expert 1 might specialize in nouns and entities
  • Expert 2 might specialize in verbs and actions
  • Expert 3 might specialize in emotional/descriptive content

Why? Gradient descent naturally finds the path of least resistance. If expert 0 starts becoming slightly better at function words (by chance), the router sends more function words to it → it gets more training signal for function words → it gets better → more are routed there. Positive feedback loop.

The Inference Win

Full GPT-OSS 20B:

Total parameters: 20 billion

Active parameters per token: 3.6 billion (only 18%!)

A 20B parameter model that runs as fast as a 3.6B model. You get the knowledge capacity of 20B (because different tokens use different experts, so total knowledge stored = 20B) with the speed of 3.6B (because any single token only passes through 3.6B weights).

What if an expert goes dormant?

Problem: If the router stops sending tokens to an expert, that expert gets no gradients → stops learning → becomes even less useful → router sends even fewer tokens. "Expert collapse."

Mitigation strategies (used in production):

  • Load balancing loss: penalty if experts receive very unequal traffic
  • Random routing: occasionally route tokens to non-top-K experts for exploration
  • Expert pruning: remove consistently dormant experts, add new random ones

In our nano version (4 experts, top-2): expert collapse is rare because every expert gets 50% of tokens on average.

Deep Question: Why Do Attention Sinks Exist?

The Question

Why does the model pay high attention to early tokens (even when irrelevant)? Why can't it just learn to attend uniformly when it has no preference? What does the sink actually fix?

The Softmax Constraint

# Softmax MUST sum to 1.0 across all positions
attention_weights = softmax([0.2, 0.3, 0.1, ...])  # Must sum to 1.0

# Problem: what if a token genuinely has NOTHING to attend to?
# Example: "The" at the beginning of a new paragraph
# It has no prior context, but softmax FORCES it to distribute 1.0 somewhere

# The model's solution: dump excess attention on position 0
# Token 0 becomes a "garbage collector" for homeless attention

The Sliding Window Problem

# Full causal attention: token at position 500 can dump to position 0
# ✓ Token 0 absorbs excess attention harmlessly

# Sliding window (w=128): token 500 can only see positions 372-499
# ✗ Position 0 is not accessible!
# ✗ Excess attention spills onto random positions within the window
# ✗ This CORRUPTS the attention pattern for those random positions

Without sinks, sliding window attention degrades because there's no safe place to dump excess probability mass.

How the Sink Fixes This

# Standard attention:
weights = softmax([Q·K₁, Q·K₂, ..., Q·Kₙ])
#         ←── distributes 1.0 across real tokens ──→

# With sink (one learnable parameter per head):
weights = softmax([Q·K₁, Q·K₂, ..., Q·Kₙ, sink_bias])
#         ←── distributes 1.0 across real tokens + 1 virtual slot ──→

# After softmax, DROP the sink column:
weights = weights[:-1]   # Only use weights for real tokens
# Now weights sum to LESS than 1.0 — the "missing" probability went to sink

Effect: The model can now "choose" to attend to nothing (by putting probability mass on the sink). Real attention scores stay clean and meaningful.

Cost: One learnable scalar per attention head. 16 parameters total. Negligible.

Analogy: The "Abstain" Option

Imagine a multiple-choice test where you MUST circle an answer (softmax = must sum to 1). Even when you don't know, you're forced to guess — sometimes corrupting your other answers by over-thinking.

Attention sinks = adding an "I don't know" option. Now when you're uncertain, you circle "IDK" and your real answers on other questions stay clean.

Deep Question: Why Residual Connections?

The Question

Why output = x + layer(x) instead of just output = layer(x)? What happens without residuals in a 12-layer transformer?

The Vanishing Gradient Problem

# Without residuals: output = f₁₂(f₁₁(...f₂(f₁(x))...))
# Gradient at layer 1: ∂loss/∂θ₁ = ∂loss/∂f₁₂ × ∂f₁₂/∂f₁₁ × ... × ∂f₂/∂f₁ × ∂f₁/∂θ₁
# = product of 12 Jacobians
# If each |∂fᵢ/∂fᵢ₋₁| < 1 → product → 0 (vanishing)
# If each |∂fᵢ/∂fᵢ₋₁| > 1 → product → ∞ (exploding)

# With residuals: output = x + f₁₂(x + f₁₁(x + ...f₁(x)...))
# Gradient: ∂loss/∂x includes a DIRECT path (through all the additions)
# ∂(x + f(x))/∂x = 1 + ∂f(x)/∂x
# The "1" ensures gradient never vanishes completely!

The "Information Highway" Interpretation

Without residuals: each layer OVERWRITES the previous representation. If layer 5 makes a mistake, all downstream layers must compensate. The input signal is gone.

With residuals: each layer ADDS information to an existing stream. The original signal is always present. Layer 5's mistake only adds noise — the clean signal from layers 1-4 still flows through.

Analogy: Without residuals = rewriting a document from scratch at each edit. With residuals = making tracked changes — the original is always recoverable.

Why this enables depth

Residual connections make each layer learn a delta (change), not a full transformation:

# Layer output = input + small_change
# The network learns: "given this context, what should I ADD?"
# Not: "given this context, what should the entire representation BE?"

Learning small changes (refinements) is much easier than learning entire representations. This is why transformers can be 100+ layers deep while vanilla networks struggle beyond 10.

Deep Question: Why Pre-Norm Over Post-Norm?

The Question

GPT-2 normalizes AFTER the attention/FFN (post-norm). GPT-OSS normalizes BEFORE (pre-norm). Why? What's the difference?

The Two Approaches

# Post-norm (GPT-2, original Transformer):
x = norm(x + attention(x))    # normalize AFTER adding residual
x = norm(x + ffn(x))

# Pre-norm (GPT-OSS, LLaMA, modern LLMs):
x = x + attention(norm(x))    # normalize BEFORE the sublayer
x = x + ffn(norm(x))

Why Pre-Norm Wins

AspectPost-NormPre-Norm
Gradient flowNorm in residual path (dampens gradient)Clean residual path (gradient flows freely)
Training stabilityNeeds careful warmup + LR tuningMore forgiving, trains easily
Deep networksUnstable beyond ~12 layers without tricksStable at 100+ layers
Final qualitySlightly better IF you can stabilize it~Equal with much less effort

The Key Difference: Where does the gradient flow?

Post-norm: x = norm(x + sublayer(x)) — the gradient from loss must pass THROUGH the norm to reach earlier layers. Norm's jacobian can suppress gradients.

Pre-norm: x = x + sublayer(norm(x)) — the residual x = x + ... provides a clean shortcut for gradients. They bypass the norm entirely via the addition. This is why pre-norm trains more stably.

Deep Question: Why Cosine LR Schedule?

The Question

Why not just use a constant learning rate? Or linear decay? Why specifically cosine? And why warmup first?

Why Not Constant LR?

Early training: Weights are random. Loss landscape is rough with many local minima. High LR helps escape bad regions and find good basins.

Late training: Weights are near optimal. Same high LR causes the model to bounce over the minimum instead of settling into it. Need low LR for fine-grained convergence.

Constant LR = either too slow early or too jittery late. Can't be optimal for both phases.

Why Cosine Over Linear Decay?

# Linear decay: LR drops at constant rate
# Problem: spends too little time at high LR (exploration phase)
#          and too much time at medium LR (no man's land)

# Cosine decay: LR stays high longer, then drops steeply at the end
# Benefit: maximum time in "exploration" mode
#          sharp transition to "exploitation" mode
#          smooth curve → no sudden training dynamics shifts

Cosine decay approximately follows the optimal schedule predicted by learning theory: spend most of your compute budget exploring, then rapidly converge at the end.

Why Warmup?

At step 0: Weights are random. Gradients are HUGE and point in essentially random directions. A large LR step from random gradients = catastrophic parameter updates that are very hard to recover from.

Warmup (200 steps): Start with LR = 3e-5 (10× smaller than peak). Gradually increase to 3e-4 over 200 steps. This gives the model time to "orient itself" — find the rough direction of the loss landscape before taking large steps.

Analogy: Warmup is like slowly accelerating a car. Slamming the gas from standstill on a slippery road (random weights) = spin out. Gentle acceleration first = maintain control.
